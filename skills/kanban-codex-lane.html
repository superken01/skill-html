Kanban Codex Lane

讓 Hermes Kanban worker 把被指派的卡片轉交給 Codex CLI 執行。

中文整理

這個技能描述在 Hermes Kanban 架構中，如何把某張卡片交給 Codex CLI lane。Kanban 負責任務狀態、指派與留言；Codex 負責在 repo 中實作。關鍵是 worker 必須先讀卡片、確認範圍，再把足夠上下文轉成 Codex 任務，最後把測試結果與阻塞原因回寫到 kanban。

何時使用

  • 任務已經在 Hermes Kanban board 上，並且適合交給 Codex 做 repo 內修改。
  • 需要多個 worker 平行處理不同卡片，同時保留任務狀態與審計紀錄。
  • 希望 Codex lane 只處理程式碼實作，協調與驗收仍由 Kanban/orchestrator 管理。

不建議使用情境

  • 卡片沒有明確 acceptance criteria。
  • 任務需要跨多個 repo 或人工產品決策，Codex lane 無法獨立完成。
  • 工作區沒有乾淨 branch/worktree，可能與其他 worker 衝突。

建議流程

  • 讀取目前 kanban task：了解目標、連結、留言、阻塞狀態與驗收條件。
  • 檢查 repo 與 branch：確保 worker 在正確工作目錄，並確認沒有未預期變更。
  • 組裝 Codex prompt：包含卡片內容、可改範圍、測試命令、禁止事項、回報格式。
  • 執行 Codex CLI：讓 Codex 實作、跑測試、輸出摘要。
  • 審核 diff 與測試：worker 必須自己檢查，不可只轉貼 Codex 的成功訊息。
  • 更新 kanban：成功就附上 commit/測試結果並 complete；失敗就 block，寫清楚缺什麼。

命令範例

Codex lane 任務執行範例

codex exec --full-auto "你是 kanban worker。請依照卡片內容完成實作，限制只改相關檔案，跑測試並回報結果。"

完成後檢查變更

git status --short && git diff --stat

常見驗證命令樣式

python -m pytest tests/ -q

注意事項

  • Kanban 狀態是事實來源；Codex 只是執行器。任何成功、失敗、阻塞都要回寫卡片。
  • 不要讓 Codex 自行擴大範圍。prompt 要明確列出不應修改的區域。
  • 若 Codex 產生不可信變更，worker 應停止並 block，而不是硬 merge。
