01

總覽：Hermes 擁有任務，Codex 是輸入 lane

對應來源章節：Overview

這個 skill 定義輕量 Hermes + Codex 雙 lane 慣例。Hermes worker 永遠是任務 owner：讀取 Kanban 卡片、判斷 Codex 是否適合、建立或選擇隔離 workspace、啟動與監控 Codex、檢查 diff、執行驗證，最後由 Hermes 回寫 complete 或 block。

不要誤解

Codex output 不是 task completion signal、不是 trusted reviewer，也不應直接寫入 durable Kanban state。

02

何時使用 / 何時不要使用

對應來源章節：When to Use

  • Kanban task 是 coding、refactor、documentation、test 或 mechanical migration。
  • 有清楚 acceptance criteria。
  • Hermes 可以在一個 run 內評估 bounded diff。
  • repo 可以複製或建立隔離 git worktree/branch。
  • Codex 結束後，Hermes 可以自行跑相關測試。
  • prompt 能寫清楚安全限制和不得變更的檔案。
  • 不要在需要未記錄的人類判斷時使用。
  • 不要在 worker 缺 repo access、Codex auth 或沒有時間 reconcile 時使用。
  • 不要碰 secrets、credential stores、private user data 或 production order-entry systems。
  • 小型直接修改更快更安全時不要啟動另一個代理。
  • research-only 任務應產出 handoff，而不是 diff。
  • 絕不可只根據 Codex 自述就標 Done。
03

所有權規則

對應來源章節：Ownership Rules

  1. Hermes 擁有 Kanban lifecycle；Codex 不得代替 worker 呼叫 complete/block/create 或 board CLI。
  2. Hermes 擁有最終接受權；Codex commits/diffs 在 review 和 verify 前都視為 untrusted patches。
  3. Hermes 擁有測試；Codex 可跑測試，但只是 advisory，Hermes 必須重跑 canonical verification。
  4. Hermes 擁有安全；若 Codex 變更 safety boundaries、risk gates、live trading behavior 或 secrets handling，即使測試過也拒絕。
  5. Hermes 擁有清理；卡住的 Codex process 和臨時 worktree 都要處理。
04

必要 worktree 與 branch pattern

對應來源章節：Required Worktree and Branch Pattern

不要在共享的 dirty checkout 直接跑 Codex。branch/worktree 名稱要能追溯到 Kanban task，讓未信任 edits 留在隔離區。

推薦變數與建立隔離 lane

TASK_ID="${HERMES_KANBAN_TASK:-t_manual}"
REPO="/path/to/repo"
BASE="$(git -C "$REPO" rev-parse --abbrev-ref HEAD)"
SAFE_TASK="$(printf '%s' "$TASK_ID" | tr -cd '[:alnum:]_-')"
BRANCH="codex/${SAFE_TASK}/$(date -u +%Y%m%d%H%M%S)"
WORKTREE="/tmp/${SAFE_TASK}-codex-lane"

git -C "$REPO" fetch --all --prune
git -C "$REPO" worktree add -b "$BRANCH" "$WORKTREE" "$BASE"
git -C "$WORKTREE" status --short --branch

若目前 Kanban workspace 本身已經是此任務的隔離 worktree，且 git status --short 乾淨，只能在這個安全前提下建立 sibling Codex branch。否則另建臨時 worktree，reconciliation 後再 cherry-pick 或 copy accepted commits。

05

Codex 能力檢查與模式選擇

對應來源章節：Codex Capability Checks / Mode Selection

能力檢查

command -v codex
codex --version
codex features list | grep -i goals || true

需要 /goal 時才開 feature

codex features enable goals || true
codex --enable goals --version

bounded one-shot edits 用 codex exec。更廣、需要 durable objective tracking 的任務才使用 Codex /goal。Prediction-market-bot 或安全敏感 repo 不要用 --yolo；優先在隔離 worktree 中用 --full-auto，再由 Hermes reconcile。

06

Prompt 建構

對應來源章節：Prompt Construction

  • 包含 task_id、標題與完整 Kanban acceptance criteria。
  • 包含 repo path、worktree path、branch name 與 allowed file scope。
  • 明確寫入：Hermes owns Kanban lifecycle；Codex is an input lane only。
  • 要求輸出 concise summary、files changed、commits、tests run、known risks。
  • 禁止 secrets access、external messaging、board mutation、unrelated refactors、非必要 dependency upgrades。
  • 列出 Codex 可跑的驗證命令與 Hermes 之後會跑的命令。
PMB 安全限制

若是 prediction-market-bot，prompt 必須保留 paper-only、禁止 market orders、禁止 bypass price/risk checks、禁止 fake fills/PnL/order states、不得 weaken risk gates/kill switches、不得讀寫 secrets 等 safety constraints。

07

監控、逾時與 kill 條件

對應來源章節：Monitoring, Timeout, and Kill Behavior

背景啟動與監控

result = terminal(
    command="codex exec --full-auto '$(cat /tmp/codex_prompt.md)'",
    workdir=WORKTREE,
    background=True,
    pty=True,
    notify_on_complete=True,
)
session_id = result["session_id"]
process(action="poll", session_id=session_id)
process(action="log", session_id=session_id, limit=200)
process(action="wait", session_id=session_id, timeout=300)
  • 超過兩分鐘的 lane，定期送 Kanban heartbeat。
  • 若 Codex 要求 secrets、production credentials 或 external permissions，立即 kill。
  • 若它修改 worktree 外檔案、做無關重寫、剩餘 runtime 不足或沒有有用輸出，也應 kill。
  • kill 後檢查 git status --short，只保留安全且有用的 patch，並記錄 rejected/timed_out 原因。
08

Reconciliation 與完成 metadata

對應來源章節：Reconciliation Checklist / kanban_complete Metadata Schema

  • git -C <WORKTREE> status --short --branch 只顯示預期檔案。
  • Hermes 已 review diff --stat 與 git diff。
  • 沒有 secrets、credentials、generated caches、無關資料或 local artifacts。
  • 安全約束仍保留。
  • Codex commits 足夠小，可以 cherry-pick 或 squash。
  • Hermes 以 repo canonical wrapper 重跑測試。
  • Codex-run tests 與 Hermes-run tests 分開列。
  • accepted diff 已套用到 Hermes-owned workspace/branch。
  • rejected/partial work 有具體理由與 artifact path。

kanban_complete.metadata.codex_lane 需描述 used、mode、worktree、branch、command、result、accepted_commits、rejected_reason、tests_run 與 artifacts。若跳過 Codex，也要用 used: false 和具體 skipped/rejected reason 交代。

09

常見陷阱與驗證清單

對應來源章節：Common Pitfalls / Verification Checklist

  • 把 Codex 自述當驗證。
  • 在使用者 dirty main checkout 裡跑 Codex。
  • 讓 Codex 擁有 Kanban lifecycle。
  • 忘記在 prompt 中放 safety invariants。
  • 快速 edits 卻使用 /goal。
  • kill 後沒有記錄原因。
  • 因為 tests pass 就接受無關 cleanup。
  • Codex 已檢查或明確 skipped。
  • 只在隔離 worktree/branch 跑。
  • prompt 包含 scope、ownership、safety、verification commands。
  • Hermes 已 review diff 與安全敏感檔案。
  • Hermes 已獨立跑 canonical tests。
  • metadata.codex_lane 符合 schema。
  • 臨時 process/worktree 已清理或記錄為 artifact。
