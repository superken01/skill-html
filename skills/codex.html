01

用途與定位

對應來源章節：Codex CLI / When to use

Codex CLI 是 OpenAI 的自主 coding agent CLI。這個 skill 的用途是：當任務已經足夠明確，而且目前工作區是一個 git repository 時，透過 Hermes terminal 將可驗證的程式任務委派給 Codex。

  • 建立功能。
  • 重構既有程式。
  • 進行 PR review。
  • 批次修正多個 issue。
核心原則

Codex 的輸出是候選變更，不是完成聲明；最後仍要由目前代理檢查 diff、跑測試、決定是否採納。

02

必要條件

對應來源章節：Prerequisites

  • 已安裝 Codex：npm install -g @openai/codex。
  • OpenAI 認證已設定；可以是 OPENAI_API_KEY，也可以是 Codex CLI login 產生的 OAuth credentials。
  • 必須在 git repository 裡執行；Codex 不接受非 git 目錄。
  • Hermes terminal 呼叫 Codex 時使用 pty=true，因為 Codex 是互動式終端程式。

Hermes 自己的 openai-codex provider 會使用 Hermes 管理的 OAuth；獨立 Codex CLI 可能使用 ~/.codex/auth.json。不要只因為缺少 OPENAI_API_KEY 就判定 Codex 無法登入。

03

一次性任務與 scratch 工作

對應來源章節：One-Shot Tasks

短任務優先使用 codex exec，讓 Codex 在完成後自行結束。若只是臨時探索，也要建立臨時 git repo，因為 Codex 需要 git repository。

在專案中執行一次性任務

terminal(command="codex exec 'Add dark mode toggle to settings'", workdir="~/project", pty=true)

scratch 目錄仍先 git init

terminal(command="cd $(mktemp -d) && git init && codex exec 'Build a snake game in Python'", pty=true)
04

長任務背景模式

對應來源章節：Background Mode (Long Tasks)

  1. 用 background=true 和 pty=true 啟動 Codex 長任務。
  2. 保存回傳的 session_id。
  3. 用 process(action="poll") 或 process(action="log") 檢查進度。
  4. 如果 Codex 提問，用 process(action="submit") 回覆。
  5. 需要停止時，用 process(action="kill")。

背景啟動與監控樣式

terminal(command="codex exec --full-auto 'Refactor the auth module'", workdir="~/project", background=true, pty=true)
process(action="poll", session_id="<id>")
process(action="log", session_id="<id>")
process(action="submit", session_id="<id>", data="yes")
process(action="kill", session_id="<id>")
05

重要 flags 與 Gateway 沙盒注意事項

對應來源章節：Key Flags / Hermes Gateway Caveat

Flag用途
exec "prompt"一次性執行，完成後離開。
--full-auto在 sandbox 中自動批准 workspace 內的檔案修改。
--yolo不使用 sandbox、略過 approval；最快但風險最高。
--sandbox danger-full-access當 Hermes gateway/service context 讓 workspace-write sandbox 失敗時使用。

在 Telegram/gateway 這類 service context 呼叫 Codex 時，bubblewrap 或 user namespace 可能因權限不足而失敗。若遇到 uid map 或 loopback 類錯誤，可改用 --sandbox danger-full-access，但要用乾淨 git status、明確 workdir、窄範圍 prompt、diff review 與測試作為安全邊界。

06

PR review、worktree 平行修正與批次 review

對應來源章節：PR Reviews / Parallel Issue Fixing with Worktrees / Batch PR Reviews

  • PR review：clone 到暫存目錄、checkout PR，再讓 Codex review against base branch。
  • 平行 issue fixing：為每個 issue 建立獨立 worktree 與 branch，各自啟動 Codex，完成後推 branch 開 PR。
  • 批次 PR review：fetch PR refs，針對每個 PR 以背景 Codex 任務平行 review，最後再貼回評論。

安全 PR review clone

REVIEW=$(mktemp -d) && git clone https://github.com/user/repo.git $REVIEW && cd $REVIEW && gh pr checkout 42 && codex review --base origin/main

建立隔離 worktree

git worktree add -b fix/issue-78 /tmp/issue-78 main

fetch PR refs

git fetch origin '+refs/pull/*/head:refs/remotes/origin/pr/*'
07

操作規則

對應來源章節：Rules

  • 一定使用 pty=true。
  • 一定在 git repo 內執行；scratch 也先 git init。
  • 短任務用 codex exec。
  • 需要實作時可用 --full-auto，但仍要檢查 diff。
  • 長任務用 background 並監控 process。
  • 不要頻繁干擾 Codex；只 poll/log 或必要時送入回覆。
  • 多個 Codex 任務可以平行，但應用 worktree 或獨立 branch 隔離。
