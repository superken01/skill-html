01

總覽

對應來源章節：Overview

Kanban Operations 是 Hermes Kanban orchestrator、worker、operator 知識的 umbrella。當工作涉及 task decomposition、profile routing、card lifecycle、blocked/retry recovery、dispatcher debugging 或 worker handoff conventions 時載入。

02

Orchestrator lane

對應來源章節：Orchestrator lane

建立卡片前，先用 hermes profile list 或詢問使用者來探索實際存在的 profiles。可平行的工作拆成獨立卡片；只有真正有依賴時才建立 parent links。

  • 不要發明不存在的 assignee。
  • acting as orchestrator 時不要自己偷做 implementation。
  • 卡片要寫清楚目標、範圍、驗收條件、禁止事項與建議驗證。
  • 支援檔：references/kanban-orchestrator/dispatch-and-timeout-debugging.md。

查看可用 profiles

hermes profile list

查看卡片列表

hermes kanban list
03

Worker lane

對應來源章節：Worker lane

Worker 接到卡片後，要先 orient on the card：讀卡片內容、先前 runs/comments，確認自己在 $HERMES_KANBAN_WORKSPACE 中工作。heartbeat 只用於有意義進度，不要當作假進度。

  • 完成時回寫 structured metadata、artifact、測試輸出與必要檔案路徑。
  • 若重大程式變更需要人工 approval，使用 review-required blocking。
  • 缺少資料、權限或外部依賴時 block 並寫明原因。
  • 支援檔：references/kanban-worker/model-stall-diagnostics.md。

Worker 確認工作區

pwd && echo "$HERMES_KANBAN_WORKSPACE" && git status --short

查看單一卡片細節

hermes kanban show <card-id>

查看執行紀錄

hermes kanban runs <card-id>
04

Operator lane

對應來源章節：Operator lane

當被問到卡片是否有跑、為何卡住、能不能重試時，回答必須基於 board evidence：list、show、runs、logs 與 dispatcher output。要區分 task creation、worker spawn、model stall、external tool failure、timeout、human block。

  1. 先用 show 與 runs/logs 確認阻塞原因。
  2. 若需要重試，先 unblock 並留下 reason。
  3. 只 dispatch 一輪。
  4. 再用 show 與 runs 驗證狀態是否改變。

查看 logs

hermes kanban logs <card-id>

解除阻塞並留下原因

hermes kanban unblock <card-id> --reason "補齊缺少資訊後重試"

派送一輪

hermes kanban dispatch --once
05

驗證清單

對應來源章節：Verification Checklist

  • 指派前已確認 profiles 存在。
  • 依賴用 parent links 表示，而不是只寫在 prose。
  • Worker summaries 包含 structured metadata 與 artifacts。
  • Retry/recovery actions 已用 show + runs 驗證。
  • 任務完成聲明有 board state、comments 或 artifacts 支撐。
