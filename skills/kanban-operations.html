Kanban Operations 看板操作總覽

中文整理

用途：這個技能是 Hermes Kanban 的操作總覽，將 orchestrator lane、worker lane 與 operator lane 的責任分清楚。它不是把英文原文搬過來，而是把日常看板操作整理成手機上也容易閱讀的中文說明：如何拆卡、指派 profile、讓 worker 在正確工作區執行、用看板證據確認狀態，以及在卡住、逾時或需要重試時如何安全恢復。

何時使用

  • 老闆或團隊要用 Hermes Kanban 拆分一批工作，並把不同卡片交給適合的 profile 或 worker。
  • 需要判斷某張卡片到底是尚未派工、worker 沒啟動、模型停住、外部工具失敗、逾時，還是等待人工審核。
  • worker 完成或阻塞任務時，需要知道應該回寫哪些摘要、證據、artifact、測試結果與下一步。
  • operator 要重試、解除阻塞、查看 runs/logs，或確認 dispatcher 是否真的把卡片送出去。

不建議使用情境

  • 只是單一 repo 的小修改，直接由目前代理完成會比建立 Kanban 卡片更快。
  • 尚未知道可用 profile 或 worker 名稱；這時不要憑空指派，應先查 profile 或詢問。
  • 任務需要立即人工產品決策，沒有清楚驗收條件，worker 無法獨立完成。

建議流程

  • Orchestrator lane：先用 hermes profile list 或向使用者確認可用 profile，再把可平行處理的工作拆成獨立卡片；只有真的有依賴時才用 parent link，不要只在文字裡描述依賴。
  • Orchestrator lane：派工時寫清楚目標、範圍、驗收條件、禁止事項與建議驗證命令；身為 orchestrator 時不要自己偷做 worker 的實作，也不要發明不存在的 assignee。
  • Worker lane：接到卡片後，先讀卡片內容、先前留言與 run 紀錄，確認自己在 $HERMES_KANBAN_WORKSPACE 內工作；只有有實質進度時才 heartbeat。
  • Worker lane：完成時要回寫結構化摘要、實際驗證結果、commit/artifact 或檔案路徑；若需要人工審核或缺少資料，應 block 並寫明原因與需要誰處理。
  • Operator lane：回答卡片有沒有跑、為什麼卡住時，必須以 list、show、runs、logs 或 dispatcher 輸出為證據，不要只看卡片標題推測。
  • Operator lane：重試時先確認阻塞原因，必要時 unblock 並寫 reason，只 dispatch 一輪，然後再用 show 與 runs 驗證狀態是否真的改變。

命令範例

查看可用 Hermes profiles

hermes profile list

查看看板卡片列表

hermes kanban list

查看單一卡片細節

hermes kanban show <card-id>

查看卡片執行紀錄

hermes kanban runs <card-id>

查看卡片或 worker logs

hermes kanban logs <card-id>

解除阻塞並留下原因

hermes kanban unblock <card-id> --reason "補齊缺少資訊後重試"

派送一輪待處理卡片

hermes kanban dispatch --once

Worker 確認工作區

pwd && echo "$HERMES_KANBAN_WORKSPACE" && git status --short

注意事項

  • 驗證清單：指派前已確認 profile 存在，沒有憑空假設 assignee。
  • 驗證清單：真正的依賴已用 parent link 表示，而不是只寫在描述文字裡。
  • 驗證清單：worker 完成、阻塞或重試後，都有用 show 與 runs/logs 交叉確認。
  • 驗證清單：完成聲明有看板狀態、留言、artifact、commit 或測試輸出支撐。
  • 注意事項：Kanban board 是狀態事實來源；worker 或其他代理的口頭摘要只能當線索。
  • 注意事項：重試前先分辨是派送、啟動、模型、工具、逾時或人工審核問題，避免盲目連續 dispatch。
  • 注意事項：大量卡片平行執行時，優先隔離 workspace、branch 或 repo clone，避免 worker 互相覆蓋。
