01

總覽

對應來源章節：Overview

這是 agent delegation 與 profile setup 的 umbrella skill。當使用者要跑 coding agent、建立 AI software-agency team、設定 agent fleet 的模型/profile，或要在 Claude Code、Codex、OpenCode、Hermes subagents/Kanban 之間選擇時使用。

02

決策流程

對應來源章節：Decision Flow

  1. 先判斷工作是短期 reasoning（delegate_task）、durable cross-profile work（Kanban），還是 external agent CLI job。
  2. 指派前驗證選定 CLI 或 profile 真的存在。
  3. 傳入 self-contained context：repo path、goal、constraints、tests、branch/PR expectations、output contract。
  4. 要求 child agents 回傳可驗證 handles：file paths、diffs、test output、PR URLs、run IDs。
  5. 回報成功前，自己驗證 external side effects。
03

External coding CLIs lane

對應來源章節：External coding CLIs

當使用者明確要求某個 agent，或 repo-local CLI workflow 是正確 execution substrate 時，才使用 Claude Code、Codex 或 OpenCode。執行方式要 bounded、inspectable，並保存輸出；變更必須用 git diff/tests 驗證。

04

OpenCode API lane / API mode

對應來源章節：OpenCode API lane / OpenCode API mode

OpenCode 可不經 TUI，由 opencode serve 或 npx -y opencode-ai serve 啟動 HTTP/OpenAPI 服務；可用 /doc、/global/health、/session、/session/{sessionID}/message 等 endpoint。當 Hermes/Kanban integration 需要 resident server、session control、event/message polling 時，優先考慮這條 lane。

OpenCode 也可搭配 @opencode-ai/sdk 與 opencode run --attach 做程式化控制；如果 worker 需要 durable、可檢查的 OpenCode lane，這比單純 one-shot CLI 更合適。

05

Hermes profiles 與 model providers

對應來源章節：Hermes profiles and model providers

profile setup、custom providers、Databricks Model Serving endpoints、per-profile model routing 都走這條 lane。完成設定後，要用 hermes -p <profile> model 或最小 profile run 驗證。

06

多代理 agency teams 與 profile handoff

對應來源章節：Multi-agent agency teams

設計 team 時保持小而明確：先探索現有 profiles，再記錄 planning、implementation、review、research、operations 各由哪個 profile 負責。

當使用者要求把對話轉交給某個 team role/profile（例如 pm/Ava）時，應優先嘗試真實 profile/agent handoff：one-shot profile call、Kanban durable work，或短期 role-isolated delegate_task。聲稱 handoff 成功前，要確認 profile 能實際啟動；若 auth、rate limit 或設定阻塞，要明講並使用暫時 fallback，不可假裝 profile 已跑。
